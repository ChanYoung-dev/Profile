rpm
- 패키지를 관리하는 용도로 사용
- 로컬에 파일이 있어야 설치 가능
- 파일 형식 : 이름-버전-릴리즈 버전-아키텍처.rpm
- Windows에서 .exe 파일과 흡사
- 컴파일되어 설치할 실행파일, 설정파일, 라이브러리 등을
하나로 묶어놓은 파일

설치된 패키지를 모두 나열
# rpm -qa
- q 옵션 : 쿼리, 특정 패키지에 대한 정보를 확인할 때
사용해야 하는 필수 옵션
- qa 옵션 : 설치된 패키지를 모두 나열
- noarch : 패키지가 python, perl 같은 스크립트같은 경우
32비트, 64비트 상관 없이 동작하므로 X86_64나 i686 대신
noarch 라고 표시됨

rpm 파일 설치
# rpm -ivh httpd-2.4.6-97.el7.centos.2.x86_64.rpm
- 파일의 위치를 알아야 함

yum
- rpm 업그레이드
- rpm 기반의 시스템을 위한 자동 업데이터 및 패키지 설치, 
제거 프로그램
- rpm과 다르게 인터넷 사용 필수
- 패키지 간 의존성 문제 해결
(ex. A 패키지를 설치하기 전에 필요한 B 패키지까지 모두 
한 번에 설치 가능)

패키지 설치

(ex. yum -y install httpd)

패키지 업데이트

패키지 제거

있는 것을 확인 불가, 사용 권장 X

저장소(리포지포리)
- /etc/yum.repos.d/
# vim /etc/yum.repos.d/CentOS-Base.repo
...
[base]
name=CentOS-$releasever - Base
mirrorlist=http://mirrorlist.centos.org/?release=$releasever&arch=$basearch&repo=os&infra=$infra
#baseurl=http://mirror.centos.org/centos/$releasever/os/$basearch/
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-CentOS-7
...
- 확장자명은 무조건 .repo
- 속성
 - [ID] : 아이디 지정
 - name : 이름 지정, 시스템에서 인식(대부분 ID 부분과 동일하게 사용)
 - mirrorlist : 중간 저장소
 - baseurl : 다운로드하려는 패키지의 주소 지정
 - enabled
  - 동작 여부 설정
  - 0 또는 1, true 또는 false로 표현
   - 리포지토리를 여러 개 설정해뒀을 경우 하나라도 동작하지 
않을 경우 패키지 설치 불가
ex) 3개 설정하고 2개는 맞게, 1개 틀리게 됐을 경우 패키지 설치 X
   - 잠깐 비활성화 하는 용도로 0 설정
 - gpgcheck
  - gpgkey 사용 여부 설정
  - 0 또는 1, true 또는 false로 표현
  - gpgkey
   - gpgkey 가 0으로 설정되어 있는 경우 gpgkey 입력할 필요 X
   - 패키지를 설치하는 중에 사용할 키 값을 어디에 
저장해두었는지의 정보 표시
   - wget으로 받아옴

파일 시스템
- 파일 시스템은 해당 저장 공간에서 작업할 때 적용되는 옵션들을 
설정해두는 것(압축, 공유, 작업 설정 등)

/dev

스토리지 정보 확인

디스크 파티셔닝을 위한 선행 작업(디스크 추가)
1. 가상머신 시스템 종료
2. 가상머신 선택 후 [설정] 클릭
3. [저장소] 항목 클릭
4. [컨트롤러 : SATA] 항목 맨 왼쪽 아이콘 클릭
- 네모디스크에 "+" 모양이 합쳐진 아이콘
5. [만들기] 클릭
6. [VDI] 선택 후 [다음] 클릭
7. [동적 할당] 선택 후 [다음] 클릭
8. 용량 3GB 선택 후 [만들기] 클릭
9. 만든 하드디스크를 [선택]하여 적용

원격 파일 전송
scp
- secure copy
- ssh를 통해서 원격으로 파일을 복사할 때 사용하는 명령어
- 경로를 정확하게 알아야 함

로컬호스트의 /etc/yum.conf 파일을 student@192.168.56.11의 
/tmp 디렉토리에 복사
# scp /etc/yum.conf root@192.168.56.11:/tmp
- 확인은 192.168.56.11에서 하면 됨

sftp
- ssh처럼 접속(대화형)
- 현재 시스템이 포커스, 원격 시스템에서 작업
- 파일의 위치를 파악해서 해당 파일을 가져올 수 있음

sftp 접속
# sftp root@192.168.56.11
- 포커스 : 192.168.56.10 가상머신의 /root

192.168.56.11의 /root/anaconda-ks.cfg를 192.168.56.10의 /root
디렉토리로 복사
sftp> get anaconda-ks.cfg

복사된 파일 확인
sftp> exit
# ls
결과) anaconda-ks.cfg  dirA  fileA

sftp 접속
# sftp root@192.168.56.11

192.168.56.10의 /root/fileA를 192.168.56.11의 /root
디렉토리로 복사
sftp> put fileA
sftp> ls
결과) anaconda-ks.cfg   fileA

작업 스케줄링
at
- 단일성 작업 예약 도구
- 시스템 데몬 atd 로 프로세스 제어
- at은 백그라운드에서 동작, 제어 터미널 X

at 사용을 위한 선행작업
# yum -y install at
# systemctl status atd
# systemctl start atd
# systemctl status atd

at 를 사용한 파일 생성
# at now +1min
- 현재 시간으로부터 1분 뒤 실행
at>  echo "test" > test.txt
at> <EOT>
- <EOT> 는 직접 입력하는 것이 아닌 ctrl + d 버튼을 누르면 됨

기준 시간이 될 수 있는 문자열 at 예제
# at teatime tomorrow
- teatime은 16:00를 의미

# at noon +4 days
- 4일 뒤 정오에 실행

# at 5pm august 3 2022
- 정확한 날짜에 맞춰서 예약 가능

작업 예약 확인
# at -l
결과)
2       Tue Jan 18 16:00:00 2022 a root
3       Fri Jan 21 12:00:00 2022 a root
4       Wed Aug  3 17:00:00 2022 a root

예약 작업 제거
# atrm 2		<--- 작업 번호로 제거
# at -l
결과)
3       Fri Jan 21 12:00:00 2022 a root
4       Wed Aug  3 17:00:00 2022 a root

주기적인 작업 예약
- 예약된 작업을 반복 실행
- crond 데몬이 작업을 제어(crontab 구성 파일 해석)

crontab 파일 확인
# vim /etc/crontab
- crontab 파일에 설명을 확인

crontab 파일 예제
* * * * * root echo "test" > test.txt
- root 사용자가 echo "test" > test.txt 실행
- 매분 매시 매일 매월 매요일마다 실행   ->   1분마다 계속 실행
- 현재 시간을 기준으로 실행
 - 분 자리에 * 가 표기될 경우 현재 시간 기준

* * * * * user1 echo "test" > test.txt
- /home/user1 디렉토리에서 echo "test" > test.txt 실행
- 매분 매시 매일 매월 매요일마다 실행   ->   1분마다 계속 실행

0 1 * * * root echo "test" > test.txt
- 매일 오전 1시에 실행

0 1 * * 0 root echo "test" > test.txt
- 매월 매일 일요일 오전 1시에 실행

0 1 1 * 0 root echo "test" > test.txt
- 매월 1일 일요일 1시에 실행
- 1월 1일이 월요일? 실행 X

0 1 1 2 0 root echo "test" > test.txt
- 2월 1일 일요일 1시에 실행
- 2월 1일이 월요일? 실행 X

0/3 1 * * * root echo "test" > test.txt
- 매일 오전 1시에 3분마다 실행
- 2시가 되면 실행 X

- 작업 제거도 가능(vim 형식이기 때문)

반복 작업 예약 확인
# crontab -l

반복 작업 제거
# crontab -r
- 모든 반복 작업 제거

/etc/cron.d/
- cron.d 디렉토리 밑에 반복 작업을 분류하여 생성 가능

/etc/anacrontab
- 시스템이 꺼져 있거나 절전 모드로 인해 실행되지 않을 경우
시스템이 준비되면 작업 실행

# vim /etc/anacrontab
...
#period in days   delay in minutes   job-identifier   command
1              5       cron.daily              nice run-parts /etc/cron.daily
7              25      cron.weekly             nice run-parts /etc/cron.weekly
@monthly  45     cron.monthly            nice run-parts /etc/cron.monthly
------------  ---    ---------------           --------------------------------------
       1        2            3                                        4

1. 기간(일)
- @daily 는 정수 1과 동일, 매일 실행됨을 의미
- @weekly 는 정수 7과 동일, 매주 실행됨을 의미
- @monthly 는 매달 실행됨을 의미(달마다 일 수가 다르기 때문)

2. 지연(분)
- 시스템이 준비된 후 작업을 시작하기 전에 crond 데몬이 대기해야
하는 시간

3. 작업 식별자
- 작업이 식별되는 고유 이름

4. 명령
- 실행할 명령
