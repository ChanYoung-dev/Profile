접근 제어 리스트(ACL)
- 접근 제어를 세밀하게 하기 위해 사용
- 권한을 쓰는 이유 : 리눅스가 멀티유저 환경이므로 각 사용자별로 접근 제어를
설정해두어야 함(보안적 요소)
- chmod로 소유자나 소유그룹 같은 경우 따로 권한 설정을 할 수 있음
- ACL을 사용하는 이유 : 소유자나 소유그룹 이외에 특정 사용자/그룹에 대해
권한을 따로 설정 가능

# ls -l
결과)
-rw-------. 1 root root 1340 Jan 17 15:08 anaconda-ks.cfg
...
- 권한 뒤에 . 으로 표시되어 있으면 ACL이 설정되어 있지 않다는 의미(일반 권한)
- 권한 뒤에 + 로 표시되어 있으면 ACL이 설정되어 있다는 의미, 소유 그룹 권한이
다르게 표시될 수 있음

ACL 권한 확인 명령어
# getfacl anaconda-ks.cfg
결과)
# file: anaconda-ks.cfg
# owner: root
# group: root
user::rw-
group::---
other::---
- 파일의 세부권한 확인

ACL 파일 권한 변경
형식) setfacl [option] ENTRY:NAME:PERMS file-name
- 많이 사용하는 옵션
 - m : 추가 / 수정할 때 사용하는 옵션
  - 소유자의 권한을 바꾸고 싶을 때 u::<perm> 파일명
  - 소유그룹의 권한을 바꾸고 싶을 때 g::<perm> 파일명
  - 특정 사용자의 권한을 설정할 때 u:<user_name>:<perm> 파일명
  - 특정 그룹의 권한을 설정할 때 g:<group_name>:<perm> 파일명
  - mask의 권한 m::<perm> 파일명

실습 전 선행 작업(유저나 파일이 없을 경우)
# useradd user1
# touch anaconda-ks.cfg

anaconda-ks.cfg 파일에 특정 사용자 ACL 설정
# setfacl -m u:user1:rwx anaconda-ks.cfg
# getfacl anaconda-ks.cfg
결과)
# file: anaconda-ks.cfg
# owner: root
# group: root
user::rw-
user:user1:rwx	<--- 추가
group::---
mask::rwx		<--- 추가
other::---

mask 값
- 특정 사용자, 소유 그룹, 특정 그룹에 부여할 수 있는 최대 권한을 의미
- mask 값을 별도로 지정하지 않으면 자동으로 rwx 권한 부여

anaconda-ks.cfg 파일에 특정 그룹 ACL 설정
# setfacl -m g:user1:r-x anaconda-ks.cfg
# getfacl anaconda-ks.cfg
결과)
# file: anaconda-ks.cfg
# owner: root
# group: root
user::rw-
user:user1:rwx
group::---
group:user1:r-x	<-- 추가
mask::rwx
other::---

anaconda-ks.cfg 파일에 ACL mask 값 설정
# setfacl -m m::r-- anaconda-ks.cfg
# getfacl anaconda-ks.cfg
결과)
# file: anaconda-ks.cfg
# owner: root
# group: root
user::rw-
user:user1:rwx                  #effective:r--
group::---
group:user1:r-x                 #effective:r--
mask::r--
other::---
- mask 값을 파일 소유자, 기타 사용자의 권한을 제한하지 않음
- #effective:r-- 는 실제로 부여되는 권한(설정된 권한보다 낮을 때 표시)

anaconda-ks.cfg 파일에 setuid 설정 후 세부 권한 확인
# chmod u+s anaconda-ks.cfg
# getfacl anaconda-ks.cfg
결과)
# file: anaconda-ks.cfg
# owner: root
# group: root
# flags: s--	<-- 추가

group::---
group:user1:r-x                 #effective:r--
mask::r--
other::---

실습 전 선행 작업(디렉토리가 없을 경우)
# mkdir dirA

dirA 디렉토리에 ACL 설정
# setfacl -m u:user1:rwx dirA/
# setfacl -m g:user1:r-- dirA/
- user1 사용자에 rwx 권한 부여
- user1 그룹에 r-- 권한 부여

디렉토리 하위에 파일, 디렉토리 생성시 기본으로 적용될 ACL 지정
- 소유자의 권한을 지정하고 싶을 때 d:u:<user_name>:<perm> 디렉토리명
- 소유그룹의 권한을 지정하고 싶을 때 d:g:<user_name>:<perm> 디렉토리명
- mask 권한 d:m:<perm> 디렉토리명

시스템 성능 튜닝
tuned
- 튜닝 프로파일을 관리하는 데몬 프로세스
- 튜닝 프로파일을 이용하여 정적 및 동적으로 튜닝 조정 가능

tuned 설치
# yum -y install tuned

튜닝 프로파일 리스트 확인
# tuned-adm list

balanced : 절전과 성능 간에 절충이 필요한 시스템 적합
desktop : balanced 프로파일에서 파생. 대화형 애플리케이션의 빠른 응답 제공
throughput-performance : 최대 처리량을 얻기 위해 시스템 튜닝
latency-performance : 전력 소비가 크더라도 대기 시간이 짧아야 하는 서버 시스템
에 적합
network-latency : latency-performance 프로파일에서 파생. 낮은 네트워크 대기
시간을 제공하기 위해 사용
network-throughput : throughput-performance 프로파일에서 파생. 최대 네트워크
처리량을 얻기 위해 사용
powersave : 최대 절전을 위해 시스템을 튜닝
oracle : oracle 데이터 베이스 부하에 최적화
virtual-host : 가상 시스템의 호스트로 사용될 경우 최대 성능을 얻기 위해 사용
virtual-guest : 가상 시스템에서 실행할 경우 최대 성능을 얻기 위해 사용

tuned-adm
- tuned 데몬의 설정을 관리하는데 사용하는 명령어

현재 활성 튜닝 프로파일을 식별
# tuned-adm active
결과) Current active profile: virtual-guest

현재 활성 튜닝 프로파일을 변경
# tuned-adm profile virtual-host
# tuned-adm profile virtual-guest

시스템에 맞는 튜닝 프로파일을 권장
# tuned-adm recommend

tuned 튜닝 활동 해제
# tuned-adm off

tuned 활성화
# tuned-adm profile virtual-guest

프로세스 스케줄링 조정
Linux 및 기타 운영 체제는 타임 슬라이싱 또는 멀티태스킹이라는 기술 사용
- 보유 처리 장치보다 많은 프로세스 실행
- 운영 체제 프로세스 스케줄러는 단일 코어의 프로세스 간에 빠르게 전환되어
마치 프로세스가 동시에 실행되는 것처럼 보임

상대적인 우선 순위
- 프로세스마다 중요도가 다름
- 실행되는 프로세스마다 상대적으로 우선 순위 지정 가능

nice 수준 및 권한 설정
- nice 확인 할 때 top 명령어 사용
- nice 값이 낮을 경우 : 우선 순위 up
- nice 값이 높을 경우 : 우선 순위 down
- 일반 사용자가 nice 수준을 낮게 설정할 경우 동일한 시스템에서 실행되는 다른
프로세스에 영향이 미칠 수 있으므로 root 사용자만 프로세스 nice 수준을 낮게
설정 가능
- 일반 사용자의 경우 nice 초기 설정 할 때 0 ~ 19 설정 가능
- root 사용자의 경우 nice 초기 설정 할 때 -20 ~ 19 설정 가능
- -20 ~ 19 범위 내에서 설정 가능

nice 값 초기 설정
# nice -n 11 sleep 1000 &

renice
- nice 값을 변경(우선 순위 변경)
# renice -n -1 31989

nice 값 확인 명령어
# ps -o pid,comm,nice 31989

PR (PRIORITY)
- nice 값을 이용하지 않고, priority 값만을 따지는 프로세스들이 존재
 - 리얼 타임 프로세스
- 모든 프로세스는 priority 값을 가짐
 - 1 ~ 139 까지 범위 설정 가능
 - 1이 우선 순위가 제일 높고, 139가 우선 순위가 낮음
- 음수 값을 갖는 프로세스들은 리얼 타임 프로세스
- 양수 값을 갖는 프로세스들은 일반 프로세스
- 일반적인 공식 PR - NI = 20

스토리지
- 물리적인 저장장치(디스크)
- 컴퓨터 데이터를 저장하는 장치
- 연결 방식
 - DAS
  - Direct Access Storage
  - 직접 연결하는 방식
  - 케이블 통해서 직접 연결
 - NAS
  - Network Access Storage
  - 네트워크로 연결해서 쓰는 스토리지
  - 서비스를 제공받는 느낌
 - SAN
  - Storage Area Network
  - 네트워크로 연결해서 쓰는 스토리지
  - DAS 처럼 직접 연결해서 직접 구축하여 사용

파티션
- 하드 드라이브를 여러 개의 스토리지로 나눈 공간

파티셔닝
- 하드 드라이브를 한 개 이상의 파티션으로 나누는 기술
 - 효율적으로, 목적에 맞게 사용하기 위함

사용하고 있는 장치의 목록
# lsblk

파티셔닝 방법 2가지
- MBR
 - 파티셔닝 시 2TB까지만 지원(디스크가 2TB 넘어가면 안 됨)
 - 최대 주 파티션(primary) 갯수 제한(4개)
 - 모든 파티션을 합쳐 15개 사용 가능(안정성 보장)
  - 최소화로 만드는 것을 권장
 - 주 파티션이 1개 만들어진 상태에서 extended 파티션을 만들 경우 주 파티션은
더 이상 생성 불가

- GPT
 - 물리 디스크가 점차 커지고 있으며 MBR 파티셔닝 체계의 2TB 디스크 및
파티션 갯수 제한의 문제 해결을 위해 등장
 - 8ZB 까지 지원
 - 파티션 갯수를 128개까지 분할 가능

부팅 방식
- BIOS
 - 펌웨어 인터페이스
  - 운영체제와 컴퓨터 펌웨어 간의 통역사 같은 역할 수행
  - 펌웨어란 하드웨어와 소프트웨어의 특성 모두 갖춘 것
 - MBR 방식으로만 사용
 - 운영 체제를 가동시키기 위해 설계된 오래된 유틸리티

- UEFI
 - 최신 PC용 표준 펌웨어 인터페이스
  - PC 기술이 진보함에 따라 온도와 전력 모니터링, 원격 보안 관리, 가상화 등의
프로세서 등 BIOS 지원할 필요가 생김
 - 2TB 이상의 고용량 하드디스크 제어에 적합
 - 하드웨어를 진단하지 않고 빠른 OS 부팅 가능

파티셔닝 도구
- fdisk, gdisk, parted
- MBR, GPT 방식을 섞어서 사용할 수 없음

fdisk
- MBR 방식 전용 파티셔닝 도구

파티셔닝 방법
# fdisk /dev/sdb
- m 옵션 : 도움말
- n 옵션 : 파티션 새로 만들기
- p 옵션 : 파티션 테이블 확인
- d 옵션 : 파티션 제거
- w 옵션 : 저장
- q 옵션 : 종료
